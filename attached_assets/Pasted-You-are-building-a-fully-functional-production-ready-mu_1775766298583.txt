You are building a fully functional, production-ready, multi-user Telegram bot that acts as an autonomous DCA (Dollar-Cost Averaging) onchain agent. This is a real submission for the Zerion Frontier Hackathon. Every transaction must be real — no mocks, no simulations, no fake data, no placeholder values anywhere in the code or UI.

This bot is public-facing. Any person on Telegram can use it. Each user gets their own isolated wallet, their own DCA configs, their own policy settings, and their own transaction history. No user can ever see or touch another user's data or wallet.

---

PROJECT OVERVIEW
A public Telegram bot where any user can:
- Create a personal onchain wallet inside the bot
- Set up automatic recurring token purchases (DCA) on Base chain
- Define their own spend policies
- Execute real onchain swaps via Zerion API on a schedule
- Track their portfolio and transaction history in real time

---

TECH STACK
- Runtime: Node.js (TypeScript)
- Telegram Bot: telegraf
- Wallet layer: Fork of https://github.com/zeriontech/zerion-ai — used as the execution and wallet base for all users
- Swap routing: Zerion API only (https://developers.zerion.io)
- Scheduler: node-cron — one global scheduler that loops through all active DCA configs across all users
- Database: PostgreSQL (via prisma ORM) — for multi-user data at scale
- Encryption: All private keys stored AES-256 encrypted in the database using a master encryption key from .env. Never stored in plaintext.
- Chain: Base mainnet only (chainId: 8453) — hardcoded policy for all users, no exceptions
- Environment: All secrets via .env

DATABASE SCHEMA (Prisma)

User {
  id              String   @id @default(uuid())
  telegramId      String   @unique   // Telegram user ID
  telegramUsername String?
  walletAddress   String   @unique
  encryptedPrivateKey String        // AES-256 encrypted, never plaintext
  createdAt       DateTime @default(now())
  dcaConfigs      DcaConfig[]
  transactions    Transaction[]
}

DcaConfig {
  id              String   @id @default(uuid())
  userId          String
  user            User     @relation(fields: [userId], references: [id])
  tokenSymbol     String   // e.g. "ETH", "WBTC"
  tokenAddress    String   // real contract address on Base
  amountUsd       Float    // per-execution spend in USD
  interval        String   // "hourly", "daily", "weekly"
  dailyCapUsd     Float    // max USD in any 24h window (default: 3x amountUsd)
  slippageCap     Float    // max slippage % (default: 1.0)
  expiresAt       DateTime
  isActive        Boolean  @default(true)
  isPaused        Boolean  @default(false)
  totalSpentUsd   Float    @default(0)
  nextRunAt       DateTime
  createdAt       DateTime @default(now())
}

Transaction {
  id              String   @id @default(uuid())
  userId          String
  user            User     @relation(fields: [userId], references: [id])
  dcaConfigId     String
  tokenSymbol     String
  tokenAmount     String   // actual token amount received
  usdAmount       Float    // USD spent
  txHash          String   // real on-chain tx hash
  status          String   // "success" | "failed" | "skipped"
  skipReason      String?  // policy reason if skipped
  executedAt      DateTime @default(now())
}

USER ONBOARDING FLOW

When a new user sends /start:
1. Check if user exists in DB by telegramId
2. If new user:
   - Generate a brand new EVM wallet (address + private key) using ethers.js
   - Encrypt the private key with AES-256 using MASTER_ENCRYPTION_KEY
   - Save encrypted key and wallet address to DB
   - Send welcome message with:
     - Their new wallet address
     - A clear warning: "Fund this wallet with USDC on Base network before setting up DCA"
     - A link to Base bridge: https://bridge.base.org
     - Instructions to always keep funds small — this is a hot wallet
3. If existing user:
   - Show wallet address, live USDC balance (from Zerion API), and active DCA count

---

POLICIES (enforced per user per config — judging criteria)

All policies are checked before every swap execution. If any policy fails, the swap is skipped and the user is notified via Telegram.

1. Chain Lock
   - All swaps locked to Base (chainId: 8453). No exceptions for any user.

2. Per-Execution Spend Limit
   - Swap amount must not exceed the user's configured amountUsd by more than 0.5%.

3. Daily Spend Cap
   - Sum of all swaps for this user in the last 24 hours must not exceed dailyCapUsd.
   - Default: 3x amountUsd. User can customize via /setup.

4. Expiry Window
   - If current time is past expiresAt, mark config inactive and notify user.

5. Slippage Cap
   - If Zerion API quote shows slippage above user's slippageCap (default 1%), skip and notify.

6. Balance Check
   - Before executing, verify user's wallet has enough USDC to cover the swap.
   - If insufficient balance, skip and send: "❌ Insufficient balance. Please top up your wallet: [address]"

---

GLOBAL SCHEDULER

- Runs every minute via node-cron
- Fetches all DcaConfigs where isActive=true AND isPaused=false AND nextRunAt <= now
- For each config:
  1. Run all policy checks
  2. If any policy fails: log skip reason to Transaction table, notify user via Telegram, update nextRunAt
  3. If all pass: execute swap via Zerion API using that user's decrypted wallet
  4. On success: save transaction, notify user, update totalSpentUsd and nextRunAt
  5. On failure: log to Transaction table, notify user, update nextRunAt
- Each user's execution is fully isolated — one user's failure never affects others

---

SWAP EXECUTION FLOW (per user)

1. Decrypt user's private key using MASTER_ENCRYPTION_KEY
2. Run all policy checks
3. Call Zerion API to get a real swap quote:
   - From: USDC on Base
   - To: user's target token on Base
   - Amount: user's amountUsd in USDC
4. Validate quote: check slippage, output token, chain
5. Sign and submit transaction using forked Zerion CLI wallet layer
6. Poll for tx receipt (timeout: 3 minutes)
7. On confirmation:
   - Save to Transaction table
   - Update DcaConfig (totalSpentUsd, nextRunAt)
   - Send Telegram: "✅ DCA complete: bought [amount] [token] for $[usd]. Tx: https://basescan.org/tx/[hash]"
8. On failure or timeout:
   - Save failed transaction to DB
   - Notify user via Telegram with reason
   - Schedule next run normally — do not retry immediately

---

ZERION API USAGE
- Base URL: https://api.zerion.io
- Auth header: "Authorization: Basic <base64(ZERION_API_KEY:)>"
- Endpoints to use:
  - GET /v1/wallets/{address}/portfolio — live balances per user
  - GET /v1/fungibles/?filter[search_query]={token} — token lookup + price
  - POST /v1/swaps/quotes — get swap quote
  - POST /v1/swaps/execute — execute swap
- Never hardcode prices, balances, or token amounts. Always fetch live.

SECURITY REQUIREMENTS
- Private keys are NEVER shown to the user under any circumstances
- Private keys are NEVER logged to console or files
- Private keys are stored AES-256 encrypted — decrypted only in memory at swap time, immediately discarded after
- Each user can only access their own data — all DB queries must filter by telegramId
- .env must be in .gitignore from the first commit
- Rate limit bot commands: max 10 commands per user per minute to prevent abuse
- Validate all user inputs before processing
